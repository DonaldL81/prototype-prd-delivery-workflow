# 产品原型开发协作流程规范

## 0. 执行版检查卡

实际开发时优先执行本检查卡。后续详细章节作为规则说明和异常情况依据。

### 0.1 启动前必须完成

1. 阅读本流程文件。
2. 收集需求概括、需求背景、业务背景和系统框架。
3. 整理并确认 `docs/project-brief.md`。
4. 确认技术栈、运行方式、构建方式和交付形式。
5. 写入开发规范：目录结构、命名、样式、Mock、演示逻辑标注、响应式和状态处理。
6. 创建 `docs/PRD.md` 骨架，并建立“PRD 待写入清单”。
7. 输出开发计划，用户确认后才能进入页面开发。

### 0.2 每个页面必须执行

1. 询问是否有参考资料：截图、HTML、线上系统、竞品、Figma、接口文档。
2. 有参考资料时先解析，再确认采用与不采用内容。
3. 输出页面开发确认卡。
4. 用户确认字段、交互、接口、后端、数据来源和验收标准。
5. 开发页面并提供实时预览。
6. 更新 `docs/page-specs.md` 和 `docs/decision-log.md`。
7. 分开记录正式开发逻辑和原型演示逻辑。
8. 将已确认的正式逻辑增量写入 `docs/PRD.md`，不得等到项目最后再一次性回忆整理。

### 0.3 遇到问题必须记录

以下情况解决后立即写入 `docs/decision-log.md`：
- 需求歧义。
- 参考资料与当前需求不一致。
- 页面字段、交互、接口或后端逻辑变更。
- 演示逻辑容易被误解为正式逻辑。
- Bug、构建问题、样式问题、数据问题。
- 方案调整或经验教训。

累计超过 5 条未整理问题时，先整理再继续开发。

### 0.4 交付前必须检查

1. 页面均可预览。
2. `dist/` 已生成。
3. 单文件 HTML 已生成。
4. 页面规格和关键决策已沉淀。
5. `docs/page-specs.md`、`docs/decision-log.md` 与 `docs/PRD.md` 已做覆盖检查。
6. PRD 已包含前端功能、交互、接口、后端、数据来源、权限、状态、异常和验收标准。
7. 演示逻辑未混入正式 PRD 逻辑。
8. 已询问是否需要写入 Figma。

### 0.5 上下文恢复必须执行

上下文过长、对话压缩、阶段不清楚、长时间中断或无法确认下一步时，以项目根目录 `AGENTS.md` 的“失忆恢复规则”为准。

恢复后必须先输出：
- 当前阶段
- 已完成事项
- 下一步
- 待确认问题

## 1. 文档目的

本文档用于约束产品原型开发过程中的协作方式，确保每个需求都按照“先理解、定技术、写规范、再规划、逐页确认、边开发边沉淀、最后交付”的方式推进。

后续每个项目开始前，必须先阅读并遵守本文档。除非用户明确调整流程，否则不得跳过关键阶段直接进入开发。

## 2. 核心原则

1. 不直接开发：在项目框架、技术栈、开发规范和开发计划确认前，不直接编写页面代码。
2. 先全局后细节：前期只确认需求背景、业务流程、系统框架和页面范围，不提前纠缠页面字段细节。
3. 技术先确认：开发前必须确认技术栈、运行方式、构建方式、目录结构、代码规范、命名规范、样式规范和 Mock 规则。
4. 逐页确认：进入开发后，每个页面都必须先确认页面功能、字段、交互、接口逻辑、后端逻辑和数据来源。
5. 参考资料先解析：每个页面开发前必须询问是否有原型截图、原型 HTML、线上系统截图、竞品截图、Figma、接口文档等参考资料。
6. 演示逻辑与正式逻辑分离：写死数据、Mock 接口、固定跳转等只作为原型演示逻辑，不写入正式 PRD 的真实开发逻辑。
7. 过程持续沉淀：已确认的业务逻辑、页面逻辑、问题和解决方案必须及时写入项目文档，避免长对话导致遗漏。
8. 交付前统一检查：交付前必须检查页面、构建产物、单文件 HTML、PRD、演示逻辑剥离和 Figma 可选写入确认。

## 3. 项目文档结构

为避免文档过多导致维护分散，默认只维护以下文档。

### 3.1 流程规范

文件：`docs/00_workflow.md`

用途：
- 固定协作流程。
- 约束阶段推进规则。
- 作为后续项目启动前的流程依据。

### 3.2 项目总控文档

文件：`docs/project-brief.md`

用途：
- 记录项目背景、业务目标、系统框架、页面清单、开发计划、当前阶段和交付要求。
- 合并项目框架、开发计划、项目状态和交付检查内容。

建议包含：
- 当前阶段
- 需求概括
- 需求背景
- 业务背景
- 业务目标
- 目标用户
- 用户角色
- 核心业务流程
- 系统数据交互流程
- 前端功能框架
- 系统模块划分
- 页面清单
- 开发顺序
- 技术栈选择
- 开发规范
- 目录结构
- 运行与构建命令
- 数据来源
- 后端能力假设
- Mock 数据规划
- 交付要求
- Figma 是否需要写入：待确认 / 是 / 否
- 待确认问题

### 3.3 页面规格文档

文件：`docs/page-specs.md`

用途：
- 按页面记录详细功能、字段、交互、接口逻辑、后端逻辑、异常状态、验收标准。
- 作为最终 PRD 的页面功能来源。

每个页面建议包含：
- 页面名称
- 页面目标
- 用户角色
- 页面入口
- 页面字段
- 页面操作
- 交互逻辑
- 接口逻辑
- 后端逻辑
- 数据来源
- 权限规则
- 加载状态
- 空状态
- 异常状态
- 验收标准
- 正式开发逻辑
- 原型演示逻辑
- PRD 写入状态：待写入 / 已写入 / 不写入
- 对应 PRD 章节

### 3.4 决策与经验记录

文件：`docs/decision-log.md`

用途：
- 记录开发过程中的需求决策、页面确认、参考资料解析、演示逻辑、问题、解决方案和经验教训。
- 每条记录必须标记是否写入 PRD。

建议包含以下区块：
- 决策记录
- 参考资料解析
- 演示逻辑记录
- 问题与经验教训

每条记录建议包含：
- 记录编号
- 时间
- 类型：业务决策 / 页面决策 / 参考资料 / 演示逻辑 / 接口逻辑 / 后端逻辑 / 问题经验
- 页面或模块
- 记录内容
- 结论
- 是否写入 PRD：是 / 否 / 待确认
- 对应 PRD 章节
- 状态：有效 / 已变更 / 已废弃 / 待确认

### 3.5 最终 PRD

文件：`docs/PRD.md`

用途：
- 作为持续更新的正式产品需求文档。
- 基于 `project-brief.md`、`page-specs.md`、`decision-log.md` 和最终页面统一整理。
- 正式 PRD 只写真实产品逻辑和真实开发逻辑，不把演示用写死逻辑当成真实需求。
- 项目启动时先创建骨架，逐页开发过程中增量写入，交付前做最终定稿。
- 每条正式逻辑必须能追溯到页面规格、业务决策或用户确认。

## 4. 标准工作阶段

### 阶段一：项目输入

用户提供需求与系统框架信息。

需求与业务输入：
- 需求概括
- 需求背景
- 业务背景
- 业务目标
- 目标用户
- 使用场景
- 约束条件
- 参考系统或竞品

系统框架输入：
- 业务流程
- 系统数据交互流程
- 前端功能框架
- 用户角色
- 核心模块
- 数据来源
- 是否需要后台
- 是否有真实接口

本阶段输出：
- 初步理解
- 关键待确认问题
- `project-brief.md` 初稿
- `PRD.md` 初始骨架

禁止事项：
- 不直接开发页面。
- 不提前深入单个页面字段细节，除非该细节会影响项目框架。
- 不把未确认的猜测写入 PRD 正式章节；未确认内容只能写入待确认问题。

### 阶段二：项目框架整理

根据用户输入整理项目骨架。

必须整理：
- 项目目标
- 用户角色
- 核心业务流程
- 系统模块划分
- 前端页面框架
- 数据流转关系
- 后端能力假设
- 本期范围
- 暂不处理内容
- 待确认问题

本阶段输出：
- `project-brief.md` 更新
- 项目框架理解稿

退出条件：
- 项目目标已确认。
- 业务流程已确认。
- 用户角色已确认。
- 系统模块已确认。
- 前端页面范围已确认。
- 待确认问题已记录。

### 阶段三：技术栈确认与开发规范

进入开发规划前，必须先确认技术方案和开发规范。

技术栈确认必须包含：
- 前端框架：如 React、Vue、原生 HTML/CSS/JS 等。
- 构建工具：如 Vite、Webpack 或无需构建的静态页面方案。
- UI 方案：如自定义样式、现有组件库、图标库、设计系统。
- 样式方案：如 CSS、SCSS、Tailwind、CSS Modules 等。
- 状态管理：如本地状态、Pinia、Redux、Zustand 等。
- 路由方案：如单页路由、多页面静态文件等。
- 数据方案：Mock 数据、静态 JSON、真实接口、接口代理。
- 运行方式：本地开发服务、静态 HTML 打开、预览端口。
- 构建产物：`dist/`、单文件 HTML、静态资源目录。
- 浏览器兼容范围。

开发规范必须包含：
- 目录结构规范。
- 文件命名规范。
- 组件命名规范。
- 样式命名规范。
- 数据 Mock 规范。
- 演示逻辑标注规范。
- 注释规范。
- 代码格式化规范。
- 页面响应式规范。
- 错误、空状态、加载状态处理规范。
- 可预览与可交付规范。

默认处理方式：
- 如果用户已指定技术栈，优先按用户指定执行。
- 如果项目已有代码，优先遵循项目现有技术栈和代码风格。
- 如果用户未指定且项目为空，默认选择适合原型交付的轻量技术栈。
- 默认不新增单独规范文档，技术栈和开发规范写入 `project-brief.md`。
- 如果项目复杂或用户要求，可以单独创建 `docs/development-guidelines.md`。

本阶段输出：
- `project-brief.md` 中的技术栈确认。
- `project-brief.md` 中的开发规范。

退出条件：
- 技术栈已确认。
- 运行方式已确认。
- 构建方式已确认。
- 目录结构已确认。
- Mock 规则已确认。
- 演示逻辑标注方式已确认。
- 开发规范已写入项目文档。

禁止事项：
- 技术栈未确认前，不输出最终开发计划。
- 开发规范未写入前，不进入页面开发。

### 阶段四：开发规划

进入开发前，必须先输出开发计划。

开发计划必须包含：
- 页面清单
- 开发顺序
- 每个页面的目标
- 前端路由规划
- 组件拆分规划
- 数据结构规划
- Mock 数据规划
- 真实接口假设
- 演示逻辑与真实逻辑区分原则
- 阶段性交付物

本阶段输出：
- `project-brief.md` 中的开发计划

退出条件：
- 用户确认开发计划。
- 页面开发顺序明确。
- 首个页面准备进入逐页确认。

禁止事项：
- 未经用户确认开发计划，不进入页面开发。

### 阶段五：逐页面确认与开发

每个页面按照固定循环推进。

#### 5.1 询问参考资料

每个页面开发前必须询问：

```text
这个页面是否有参考资料？
可以提供：原型截图、原型 HTML、线上系统截图、竞品截图、Figma 链接、现有页面代码、接口文档或字段表。
```

如果用户提供参考资料，必须先解析再开发。

参考资料解析内容：
- 参考资料类型
- 页面整体结构
- 主要功能模块
- 页面字段
- 按钮与操作
- 交互逻辑
- 数据展示规则
- 视觉布局特点
- 可复用内容
- 需要确认的问题
- 与当前需求不一致的地方
- 最终采用内容
- 未采用内容

解析结果写入 `decision-log.md`。

#### 5.2 页面开发确认卡

开发前必须输出页面开发确认卡。

模板：

```text
页面名称：
页面目标：
用户角色：
页面入口：
页面字段：
页面操作：
交互逻辑：
接口逻辑：
后端逻辑：
数据来源：
权限规则：
加载状态：
空状态：
异常状态：
正式开发逻辑：
原型演示逻辑：
验收标准：
PRD 写入状态：
对应 PRD 章节：
待确认问题：
```

用户确认后才能开发。

#### 5.3 页面开发

开发时必须遵守：
- 依据已确认的页面开发确认卡开发。
- 使用合理的 Mock 数据支持预览。
- 不把演示逻辑混入正式产品逻辑。
- 页面完成后提供实时预览。
- 用户反馈后继续调整。

#### 5.4 页面完成后的沉淀

每个页面完成后必须更新：
- `page-specs.md`：页面功能、字段、交互、接口、后端逻辑、数据来源、状态和验收标准。
- `decision-log.md`：确认过的决策、参考资料采用情况、演示逻辑、问题和解决方案。
- `PRD.md`：将本页面已确认的正式逻辑增量写入对应章节。

页面完成条件：
- 页面已开发。
- 页面可预览。
- 页面逻辑已确认。
- 正式逻辑与演示逻辑已分离。
- 相关内容已写入项目文档。
- 本页面正式逻辑已同步到 `PRD.md`，或已明确记录为不写入 PRD。
- 遇到的问题和解决方案已记录。

### 阶段六：实时预览与反馈调整

开发过程中需要提供实时预览。

要求：
- 如项目需要本地服务，启动前端服务并提供预览地址。
- 每完成一个页面，说明已完成内容和待确认内容。
- 用户反馈后，更新页面和相关文档。
- 若反馈形成明确决策，写入 `decision-log.md`。

### 阶段七：构建交付

页面开发完成后进入构建交付。

默认交付物：
- `dist/`
- 单文件 HTML 预览文件
- `docs/project-brief.md`
- `docs/page-specs.md`
- `docs/decision-log.md`
- `docs/PRD.md`

交付检查：
- `dist/` 是否生成。
- 单文件 HTML 是否生成。
- 页面是否可预览。
- 主要页面是否完成。
- 交互逻辑是否符合确认内容。
- Mock 数据和演示逻辑是否已单独记录。
- 正式 PRD 是否剥离演示逻辑。
- 数据来源是否记录。
- 接口逻辑是否记录。
- 后端逻辑是否记录。
- 是否询问需要写入 Figma。

### 阶段八：可选写入 Figma

Figma 是可选交付项，不默认执行。

交付前必须向用户确认：

```text
是否需要将本次页面或原型写入 Figma 项目？
```

如果用户确认需要，继续确认：
- Figma 文件链接
- 写入范围：全部页面 / 指定页面
- 写入方式：新增页面 / 写入已有页面
- 是否需要按当前前端页面视觉还原
- 是否需要保留组件结构

确认后再执行 Figma 写入。

如用户确认不需要，在 `project-brief.md` 或 `decision-log.md` 中记录：

```text
Figma 是否需要写入：否
```

### 阶段九：最终 PRD 定稿

PRD 不是项目最后才整理的文档，而是从项目启动开始持续维护的活文档。

维护节奏：
- 项目输入后创建 PRD 骨架。
- 项目框架确认后写入背景、目标、角色、流程和系统模块。
- 技术栈和开发规范确认后写入非功能需求、运行构建说明和交付约束。
- 每个页面确认并开发完成后，立即把正式逻辑增量写入 PRD。
- 每次逻辑变更后，同步更新页面规格、决策记录和 PRD。
- 全部页面完成后，再对 PRD 做统一校对、补齐和定稿。

PRD 默认结构：

```text
# 产品需求文档

## 1. 文档信息
## 2. 项目背景
## 3. 业务目标
## 4. 用户角色
## 5. 业务流程
## 6. 系统模块
## 7. 前端页面功能
## 8. 页面交互逻辑
## 9. 数据来源
## 10. 接口逻辑
## 11. 后端逻辑
## 12. 权限规则
## 13. 状态与异常处理
## 14. 数据字段说明
## 15. 埋点与统计
## 16. 非功能需求
## 17. 验收标准
## 18. 本期不做
## 19. 待确认问题
## 20. 原型演示说明
```

PRD 写入规则：
- 真实业务规则写入 PRD。
- 真实接口逻辑写入 PRD。
- 真实后端逻辑写入 PRD。
- 真实数据来源写入 PRD。
- 页面字段、交互、状态、权限写入 PRD。
- 演示用写死逻辑不写入正式逻辑章节。
- 演示逻辑只放入“原型演示说明”，并明确其不作为真实开发依据。

PRD 增量沉淀规则：
- 每个已确认的页面，都必须在 `PRD.md` 中有对应内容。
- 每个已确认的正式逻辑，都必须标记写入 PRD 或明确说明不写入原因。
- 每次页面完成后，先更新 `page-specs.md`，再同步更新 `PRD.md`。
- 每次用户反馈导致逻辑调整时，必须同步检查 `page-specs.md`、`decision-log.md` 和 `PRD.md` 是否都需要更新。
- 不允许只改页面代码、不改文档；也不允许只在聊天中确认、不落到文档。

PRD 覆盖检查规则：
- 交付前按页面逐项检查 `page-specs.md` 是否已进入 PRD。
- 交付前按决策逐项检查 `decision-log.md` 中“是否写入 PRD”为“是”的内容是否已进入 PRD。
- “是否写入 PRD”为“否”的内容必须说明原因，常见原因包括仅演示逻辑、本期不做、已废弃、待确认。
- 若发现 PRD 与最终页面不一致，以用户最新确认的逻辑为准，并同步修正文档和页面。

变更同步规则：
- 已确认逻辑发生变更时，必须新增一条 `decision-log.md` 记录，不直接删除旧记录。
- 被替代的旧逻辑必须标记为已废弃或已变更。
- 变更影响页面时，更新 `page-specs.md`。
- 变更影响真实产品逻辑时，更新 `PRD.md`。
- 变更只影响原型演示时，记录在演示逻辑中，不写入 PRD 正式逻辑章节。

## 5. 演示逻辑与正式逻辑区分规则

### 5.1 演示逻辑

以下内容属于演示逻辑：
- 点击登录后直接跳转首页。
- 页面使用写死账号、写死角色、写死数据。
- 前端直接模拟接口返回。
- 固定展示某些状态。
- 跳过真实权限校验。
- 跳过真实后端处理。
- 为了预览临时简化的流程。

演示逻辑记录位置：
- `decision-log.md`
- `page-specs.md` 对应页面的“原型演示逻辑”
- `PRD.md` 的“原型演示说明”

### 5.2 正式开发逻辑

以下内容属于正式开发逻辑：
- 用户输入账号密码后调用登录接口。
- 后端校验账号、密码、账号状态和权限角色。
- 接口返回 token、用户信息和权限信息。
- 页面根据接口数据渲染。
- 后端根据业务规则处理数据。
- 权限、异常、状态和数据流转按真实系统设计执行。

正式逻辑记录位置：
- `page-specs.md`
- `PRD.md`

## 6. 问题与经验教训记录规则

为避免上下文过长导致问题遗漏，开发过程中的问题必须及时闭环记录。

### 6.1 必须记录的情况

遇到以下情况必须写入 `decision-log.md`：
- 需求理解出现歧义。
- 页面字段、交互、接口逻辑发生调整。
- 参考资料和当前需求不一致。
- 原型演示逻辑和真实开发逻辑容易混淆。
- 开发中出现 bug、样式问题、构建问题、数据问题。
- 用户提出修改意见，调整后形成明确结论。
- 某个方案被放弃，改用另一个方案。
- 出现后续项目可复用的经验或注意事项。

### 6.2 触发整理机制

必须在以下节点整理问题和经验：
- 问题解决后立即记录。
- 每完成一个页面后整理一次。
- 每完成一个模块后做小复盘。
- 每次阶段切换前检查。
- 累计超过 5 条未整理问题时，先暂停继续开发，整理后再推进。

### 6.3 记录模板

```md
| 编号 | 时间 | 页面/模块 | 问题类型 | 问题描述 | 解决方案 | 是否写入 PRD | 对应 PRD 章节 | 经验教训 | 状态 |
|---|---|---|---|---|---|---|---|---|---|
| D-001 | 2026-04-28 | 登录页 | 演示逻辑 | 原型中点击登录直接跳首页，容易被误解为正式逻辑 | 在演示逻辑中记录，PRD 写真实账号校验逻辑 | 是 | 登录与权限 | 演示逻辑必须和正式逻辑分开 | 已关闭 |
```

## 7. 阶段状态规则

每次阶段性回复需要说明当前阶段。

推荐格式：

```text
当前阶段：
本次完成：
本次更新文档：
下一步：
待确认：
```

不得在未满足退出条件时跳到后续阶段。

## 8. 交付前总检查清单

交付前必须逐项检查：

- 项目框架是否已确认。
- 技术栈是否已确认。
- 开发规范是否已写入。
- 运行与构建方式是否已确认。
- 目录结构和 Mock 规则是否已确认。
- 开发计划是否已确认。
- 每个页面是否完成页面开发确认卡。
- 每个页面是否询问并解析参考资料。
- 每个页面是否可预览。
- 页面逻辑是否写入 `page-specs.md`。
- 关键决策是否写入 `decision-log.md`。
- 开发问题和解决方案是否已记录。
- 演示逻辑是否与正式逻辑分离。
- `dist/` 是否生成。
- 单文件 HTML 是否生成。
- `PRD.md` 是否定稿。
- PRD 是否包含前端页面功能。
- PRD 是否包含交互逻辑。
- PRD 是否包含接口逻辑。
- PRD 是否包含后端逻辑。
- PRD 是否包含数据来源。
- PRD 是否包含权限、状态、异常和验收标准。
- 是否询问用户需要写入 Figma。
- Figma 写入结果是否记录为待确认 / 是 / 否。

## 9. 默认执行约定

后续新项目默认按以下顺序执行：

1. 用户提供需求背景和系统框架。
2. 初始化或整理 `docs/project-brief.md`。
3. 创建或更新 `docs/PRD.md` 骨架。
4. 输出项目框架理解稿。
5. 确认技术栈。
6. 写入开发规范。
7. 输出开发计划。
8. 用户确认开发计划。
9. 按页面顺序逐页询问参考资料。
10. 解析参考资料。
11. 输出页面开发确认卡。
12. 用户确认页面开发内容。
13. 开发页面并提供实时预览。
14. 更新 `docs/page-specs.md`、`docs/decision-log.md` 和 `docs/PRD.md`。
15. 所有页面完成后构建 `dist/`。
16. 生成单文件 HTML。
17. 询问是否需要写入 Figma。
18. 定稿 `docs/PRD.md`。
19. 输出交付说明。

这套流程的目标是把产品需求、原型开发、过程决策和最终 PRD 连成闭环，确保开发过程可追溯、可复盘、可交付。
